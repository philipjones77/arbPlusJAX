# Objects Standard

Status: active

## Purpose

This document defines what belongs under `docs/objects/` and how object/catal​og
documents differ from specs, contracts, implementation notes, reports, and
status files.

It is intended to remain reusable across engineering-heavy JAX libraries.
arbPlusJAX then specializes it through its current `docs/objects/` subtree for
function catalogs and named repo object inventories.

## Scope

Apply this standard to:

- `docs/objects/`
- named runtime catalogs
- object inventories
- stable human-readable registries of important repo objects or surfaces

## Core Rule

`docs/objects/` is the home for named catalogs and object inventories.

If a document answers questions like:

- what named objects or surfaces exist?
- how are functions or object families cataloged?
- what artifacts or named object sets should a human browse in one place?

then it belongs in `docs/objects/`.

## What Belongs In `docs/objects/`

Object documents should cover:

- function catalogs
- named artifact inventories
- object-family maps
- human-readable registry-style documents
- object-set summaries that are more catalog than report

Typical document kinds:

- `function_catalog.md`
- artifact object lists
- named surface registries

## What Does Not Belong In `docs/objects/`

Do not use objects docs for:

- binding guarantees
- implementation structure explanation
- active backlog
- measured benchmark/report output
- semantic derivations or invariants

Those belong respectively in:

- `contracts/`
- `docs/implementation/`
- `docs/status/`
- `docs/reports/`
- `docs/specs/` or `docs/theory/`

## Naming Rule

Object-facing Markdown should prefer names like:

- `*_catalog.md`
- `*_inventory.md`
- `*_artifacts.md`

The section landing page remains `docs/objects/README.md`.

## Writing Rule

Object documents should:

- prioritize scanability
- use stable names and categories
- distinguish object catalogs from generated reports when both exist
- point outward to contracts/specs/theory when a listed object needs deeper
  interpretation

Object documents should not:

- quietly become a status board
- duplicate report tables line-for-line when a generated report already owns the
  current-state view
- replace specs or contracts

## Relationship To Other Doc Layers

Use this split:

- `docs/specs/`: what concepts mean
- `contracts/`: what is guaranteed
- `docs/objects/`: what named objects and catalogs exist
- `docs/implementation/`: how code is structured
- `docs/reports/`: current measured or generated state
- `docs/status/`: what remains or is underway

When documents overlap, objects docs answer “what is here to browse,” not “what
is guaranteed,” “how it works,” or “what is currently passing.”

## Indexing Rule

The objects subtree should remain browsable through:

- `docs/objects/README.md`

That README should be generated by the repo’s standard docs index generator.

## Enforceability

The repo is compliant only if:

- catalog/inventory docs live in `docs/objects/`
- object docs stay distinct from reports/status
- naming remains catalog/inventory oriented
- the generated objects index remains current

## arbPlusJAX Specialization

For this repo, `docs/objects/` is the canonical home for:

- function catalogs
- artifact/update object lists
- future human-readable registries of major public surface families
