# Specs Standard

Status: active

## Purpose

This document defines what belongs under `docs/specs/` and how specification
documents differ from contracts, objects, theory, implementation notes, reports,
and status files.

It is intended to remain reusable across engineering-heavy JAX libraries.
arbPlusJAX then specializes it through its current `docs/specs/` subtree for
semantic definitions, invariants, and capability plans-of-action.

## Scope

Apply this standard to:

- `docs/specs/`
- semantic definitions and invariants
- type/model/functionality specifications
- capability map or plan-of-action specifications when they define intended
  semantic coverage rather than implementation status

## Core Rule

`docs/specs/` is the home for semantic definitions, invariants, and intended
capability semantics.

If a document answers questions like:

- what does this concept mean in the repo?
- what semantic invariants should hold?
- how should a capability family be interpreted?
- what conceptual functionality should exist, independent of current partial
  implementation?

then it belongs in `docs/specs/`.

## What Belongs In `docs/specs/`

Specification documents should cover:

- semantic definitions
- mathematical or representational invariants
- object-model expectations
- intended capability coverage at the semantic level
- interpretation of matrix/function/operator kinds
- plans of action when they define target semantic scope rather than active
  backlog

Typical document kinds:

- semantic specification notes
- functionality specifications
- object-model specifications
- matrix/operator capability POAs used as semantic reference maps

## What Does Not Belong In `docs/specs/`

Do not use specs for:

- binding caller guarantees
- generated inventories or coverage summaries
- implementation structure notes
- practical runbooks
- active TODOs or status tracking

Those belong respectively in:

- `contracts/`
- `docs/reports/`
- `docs/implementation/`
- `docs/practical/`
- `docs/status/`

## Naming Rule

Specification-facing Markdown should prefer:

- `*_spec.md`
- `*_poa.md`

where:

- `*_spec.md` is preferred for stable semantic definitions
- `*_poa.md` is acceptable for capability/reference maps that describe intended
  semantic scope

The section landing page remains `docs/specs/README.md`.

## Writing Rule

Specification docs should:

- define concepts clearly
- state invariants explicitly
- separate semantic intent from current implementation status
- remain stable enough to guide contracts, implementation, and notebooks

Specification docs should not:

- quietly become implementation plans
- substitute for binding contracts
- track day-to-day progress

## Relationship To Other Doc Layers

Use this split:

- `docs/specs/`: semantic definitions and invariants
- `contracts/`: binding caller-visible guarantees
- `docs/objects/`: named inventories and catalogs
- `docs/theory/`: derivations and explanatory methodology
- `docs/implementation/`: code structure and implementation mapping
- `docs/reports/`: generated current state
- `docs/status/`: active backlog and completion tracking

When documents overlap, specs answer “what this means” and “what semantic scope
the repo intends,” not “what is guaranteed today” or “how it is implemented.”

## Indexing Rule

The specs subtree should remain browsable through:

- `docs/specs/README.md`

That README should be generated by the repo’s standard docs index generator.

## Enforceability

The repo is compliant only if:

- semantic definition documents live in `docs/specs/`
- specs are not silently replaced by implementation or status notes
- naming stays consistent with spec/POA intent
- the generated specs index remains current

## arbPlusJAX Specialization

For this repo, `docs/specs/` is the canonical home for:

- semantic capability maps such as structured matrix functionality
- object-model and matrix/operator interpretation documents
- future spec documents that define what repo concepts mean before or alongside
  implementation
